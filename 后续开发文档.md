# 挂机游戏后续开发文档

## 1. 项目现状

这个项目已经是一个可玩的网页挂机原型，核心方向很接近“多技能、长期成长、资源互锁”的放置 RPG。当前主要由 `index.html`、`css/style.css`、`js/*.js` 和 `config/*.json` 组成。

现有内容规模：

- 钓鱼点 11 个，鱼类 65 种。
- 伐木点 8 个，采矿点 8 个。
- 生活装备 71 件，战斗装备 46 件。
- 剧情副本 12 个，Boss 副本 3 个，无尽塔 1 个。
- 宝石采集点 3 个，宝石类型 7 种。
- 烹饪配方 27 个，卡片配方 11 个，商店永久升级 11 项。

已经具备的系统：

- 多技能挂机：钓鱼、伐木、采矿、宝石学。
- 资源生产：鱼、木材、矿石、宝石原石、战斗掉落。
- 制作成长：生活装备、战斗装备、烹饪、制卡。
- 战斗内容：剧情关卡、Boss、无尽塔、元素克制、技能、天赋。
- 长期收集：鱼类品质、图鉴里程碑、装备解锁、Boss 解锁。
- 离线收益：目前覆盖钓鱼、伐木、采矿。
- 配置外置：大量数值在 `config/*.json` 中，后续扩内容比较方便。

## 2. 目标定位

如果目标是做成类似《Melvor Idle》的游戏，建议把项目定位成：

> 一个以多技能挂机为基础，靠资源互锁、装备门槛、战斗解锁和长期收集驱动的轻量放置 RPG。

玩家循环应该更清晰地变成：

1. 选择技能挂机，获得基础资源。
2. 用资源制作工具、装备、食物、卡片。
3. 用更高效率回到采集，或进入战斗。
4. 战斗掉落稀有材料，解锁新采集点、新装备、新系统。
5. 图鉴、成就、任务、宠物、精通等长期目标持续拉长游戏寿命。

现在项目已经有这个雏形，但还缺少几个关键支柱：精通系统、任务/成就系统、清晰的解锁树、稳定的平衡工具，以及更强的离线与存档能力。

## 3. 当前主要问题

### 3.1 架构问题

`js/state.js` 承担了太多职责：默认存档、读档兼容、物品增减、装备制作、出售、升级、天气、料理、战斗、宝石、卡片等都在一个文件里。短期可以继续开发，但内容越多，越容易出现“改一个系统影响另一个系统”的问题。

建议后续逐步拆分为：

- `state/core.js`：存档结构、迁移、保存、读取。
- `systems/inventory.js`：物品增减、消耗、来源查询。
- `systems/skills.js`：通用技能经验、等级、动作结算。
- `systems/combat.js`：战斗实例、伤害、技能、掉落。
- `systems/crafting.js`：生活装备、战斗装备、卡片、烹饪的统一制作逻辑。
- `systems/offline.js`：所有技能的离线收益。

### 3.2 数据与逻辑耦合仍然偏高

虽然配置已经外置，但不少规则还是写死在代码里。例如：

- 角色总等级只取钓鱼、伐木、采矿三项最高等级，没有计入战斗、宝石、制卡。
- 离线收益目前只覆盖钓鱼、伐木、采矿，宝石、战斗、制卡等没有统一离线模型。
- `bag_capacity` 商店升级已经配置，但当前没有看到实际背包容量限制逻辑。
- Boss 解锁、采集点解锁、装备解锁有雏形，但还没有统一的“解锁条件系统”。

后续最好把“等级要求、Boss 要求、任务要求、材料要求、成就要求”抽象成通用条件。

### 3.3 长期目标不足

现在成长主要靠“升等级、换点位、做装备、打 Boss”。这适合原型，但长期留存会不够。类 Melvor 游戏的核心是每个技能都有独立长期目标，比如：

- 技能精通等级。
- 单个物品或动作的熟练度。
- 成就、任务、收藏册。
- 宠物或稀有掉落。
- 阶段性副本门槛。
- 高级资源链和消耗回收。

当前图鉴只覆盖鱼类，其他技能缺少对应收藏目标。

### 3.4 平衡风险较高

当前后期鱼、木、矿的金币和经验数值增长很快，装备和升级也有大量百分比加成。继续加系统后容易出现：

- 金币通胀，永久升级很快买满。
- 百分比叠乘导致效率爆炸。
- 高级点位收益远超低级内容，低级资源失去价值。
- 战斗属性成长和装备成长不匹配，Boss 可能突然过难或过易。

建议尽快建立平衡表或模拟脚本，计算：

- 每个等级段的升级耗时。
- 每个点位的每小时经验、金币、关键材料产量。
- 每件装备的预计制作时间。
- 每个 Boss 推荐战力和击败时间。
- 永久升级买满所需时长。

### 3.5 UI 和信息表达还可以继续增强

当前 UI 已经有完整页面，但作为长期挂机游戏，还需要更强的“决策辅助”：

- 显示每小时收益估算。
- 显示当前动作到升级还需要多久。
- 显示材料缺口和来源跳转。
- 显示下一解锁目标。
- 显示当前最优可制作装备。
- 支持一键切换目标、一键制作可升级装备。

挂机游戏玩家经常在做资源规划，界面应该主动帮玩家算。

### 3.6 存档能力偏弱

当前使用 `localStorage`，适合原型，但后续需要：

- 导出/导入存档。
- 存档版本号和迁移日志。
- 自动备份最近 N 份存档。
- 防止配置改动导致旧存档坏掉。
- 可选云存档。

类 Melvor 游戏的玩家周期长，存档安全会非常重要。

## 4. 优先开发路线

### 阶段 A：先把原型稳住

目标：让现有内容更稳定、更像正式游戏。

建议任务：

1. 增加存档版本号和迁移系统。
2. 增加导入/导出存档。
3. 修正角色总等级计算，把钓鱼、伐木、采矿、宝石、制卡、战斗都纳入。
4. 让 `bag_capacity` 真正生效，或者先从商店升级中移除。
5. 统一离线收益框架，至少加入宝石采集和无尽塔离线。
6. 增加收益预估：每小时经验、每小时金币、预计升级时间。
7. 增加材料来源面板和缺口提示。
8. 给配置增加校验，启动时检查重复 ID、缺失材料、无效配方、无效解锁。

推荐先做这些，因为它们会提高后续加内容的稳定性。

### 阶段 B：补上类 Melvor 的核心长期系统

目标：让每个技能都有长期追求。

建议新增：

1. 技能精通系统
   - 每个动作或资源有独立精通等级。
   - 精通提供小幅速度、产量、稀有率、保底等加成。
   - 满精通给图鉴/成就奖励。

2. 成就系统
   - 技能等级、资源数量、Boss 击败、装备制作、图鉴收集都可触发。
   - 成就提供少量全局加成或称号。

3. 任务系统
   - 短任务：提交资源换金币/经验。
   - 主线任务：引导玩家从钓鱼到战斗再到 Boss。
   - 周期任务：每日/每周目标。

4. 宠物或稀有收藏
   - 每个技能一个低概率宠物。
   - 宠物提供很小但永久的加成。
   - 适合长期挂机目标。

5. 技能解锁树
   - 用 Boss、等级、任务、装备共同控制新区域。
   - 避免玩家只靠金币和等级一路平推。

### 阶段 C：扩展技能矩阵

目标：让资源链更像完整放置 RPG。

推荐优先级：

1. 锻造
   - 消耗矿石制作金属锭。
   - 金属锭制作武器、防具、工具。
   - 让采矿从“直接产装备材料”变成更完整的加工链。

2. 木工/工艺
   - 消耗木材制作工具柄、弓、饰品底座。
   - 承接伐木资源。

3. 农业/草药
   - 长周期离线生产。
   - 给料理、药水、战斗消耗品提供材料。

4. 炼金
   - 把药水从商店购买改为制作。
   - 消耗草药、宝石粉、怪物材料。

5. 探险/地图
   - 战斗之外的区域解锁系统。
   - 让采集点、副本、Boss 都挂在地图节点上。

### 阶段 D：战斗深度升级

目标：让战斗不仅是数值门槛，也有构筑感。

建议：

1. 装备套装效果。
2. 武器类型：剑、弓、法杖、匕首。
3. 技能栏配置：玩家可选择自动战斗技能优先级。
4. 怪物图鉴：掉落、弱点、击杀数、首杀奖励。
5. 副本重复刷：消耗食物或门票，掉落稀有装备。
6. 战斗离线结算：按胜率、耗时、资源消耗计算。
7. 自动吃食物和补给设置。

## 5. 建议的新数据结构

### 5.1 通用解锁条件

```json
{
  "requires": [
    { "type": "skillLevel", "skill": "mining", "level": 30 },
    { "type": "bossCleared", "id": "boss_goblin_king" },
    { "type": "itemOwned", "id": "steel_pickaxe", "count": 1 }
  ]
}
```

这样钓点、矿点、配方、副本、商店物品都能共用同一套判断。

### 5.2 通用动作定义

```json
{
  "id": "mine_iron",
  "skill": "mining",
  "baseDuration": 5000,
  "xp": 25,
  "drops": [
    { "id": "iron_ore", "count": 1, "weight": 100 }
  ],
  "requires": [
    { "type": "skillLevel", "skill": "mining", "level": 10 }
  ]
}
```

后续钓鱼、伐木、采矿、采药、宝石都可以走类似结构，减少重复代码。

### 5.3 存档版本

```json
{
  "version": 2,
  "createdAt": 1710000000000,
  "updatedAt": 1710000000000,
  "state": {}
}
```

每次调整存档结构时写迁移函数，例如 `migrateV1ToV2`。

## 6. 推荐开发顺序

### 第 1 周：基础稳定

- 存档版本和导出导入。
- 配置校验脚本。
- 角色总等级修正。
- 背包容量逻辑确认。
- 宝石和无尽塔离线收益。

### 第 2 周：玩家信息增强

- 每小时收益。
- 升级剩余时间。
- 材料缺口。
- 物品来源跳转。
- 下一解锁目标。

### 第 3 周：精通和成就

- 每个采集点/鱼类/矿物的精通经验。
- 成就数据表。
- 成就奖励。
- 图鉴扩展到木材、矿石、宝石、怪物。

### 第 4 周：锻造与炼金雏形

- 矿石冶炼成锭。
- 金属锭制作战斗装备。
- 药水从商店迁移到炼金制作。
- 商店改为卖服务、图纸、稀有材料或限量补给。

## 7. 内容设计原则

1. 每个技能都要有“产出”和“消耗”。
   - 只产不耗会导致仓库堆积。
   - 只耗不产会让玩家觉得被卡。

2. 低级资源后期也要有用途。
   - 可以用于批量合成、任务提交、药水、升级消耗。
   - 不要让新手资源完全废掉。

3. 百分比加成要谨慎叠加。
   - 建议区分加法加成和乘法加成。
   - 永久加成要小，临时加成可以大。

4. 每 15 到 30 分钟应该有一个早期目标。
   - 新装备、新点位、新任务、新图鉴奖励都可以。
   - 后期目标可以逐渐拉长到数小时甚至数天。

5. 战斗应该吃生活技能资源。
   - 食物、药水、装备、卡片都应该服务战斗。
   - 战斗掉落再反过来解锁生活技能高级内容。

## 8. 技术建议

短期不一定要上框架，当前纯 HTML/CSS/JS 还能继续推进。但建议至少做这些工程化增强：

- 增加一个本地开发说明，明确需要通过 HTTP 服务运行。
- 增加配置校验脚本。
- 增加简单的模拟脚本，用来跑 1 小时、1 天、7 天收益。
- 把大文件逐步拆成系统模块。
- 所有物品 ID 建立统一索引，避免配方引用不存在的材料。
- 对 `innerHTML` 渲染的数据做转义或只渲染可信配置，防止以后导入玩家数据时产生风险。

如果后续要做得更大，可以考虑迁移到 Vite + TypeScript，但不要太早。先把游戏循环和数值模型打磨好，比先换技术栈更重要。

## 9. 下一步最值得做的 10 件事

1. 做存档导入/导出。
2. 写配置校验脚本。
3. 修正角色总等级统计。
4. 让背包扩容真的生效，或删除该升级。
5. 做每小时收益和升级剩余时间。
6. 加统一解锁条件系统。
7. 加技能精通系统。
8. 加成就系统。
9. 加锻造系统。
10. 做数值模拟表，重新平衡 1 到 100 级节奏。

## 10. 总结

这个项目不是空壳，已经有比较完整的挂机 RPG 雏形。它当前最强的地方是内容数量和系统覆盖面，最需要补的是长期目标、数值平衡、统一解锁、存档安全和代码模块化。

如果继续朝《Melvor Idle》方向升级，建议不要一开始就狂加新技能，而是先把“现有技能之间为什么互相需要”做清楚。只要资源链、战斗门槛、精通目标和离线收益稳定下来，这个项目就能从“功能很多的原型”往“能长期玩的放置游戏”迈一大步。
