# 项目技术方案

更新时间：2026-06-28

这份文档是本项目后续写代码、拆模块、加新玩法时的技术基准。之后新增系统、重构旧系统、修复杂问题时，都优先参考这里的约束。

相关文档分工：

- `项目技术方案.md`：代码结构、模块边界、后续重构顺序。以后技术实现优先参考这份。
- `后续开发文档.md`：游戏整体方向、长期系统规划、内容路线。
- `战斗系统设计文档.md`：战斗玩法规则、区域推进、装备与怪物掉落设计。

## 一、当前架构判断

项目已经从单文件原型进入了“多系统长期维护”的阶段。现在继续加玩法之前，最重要的不是马上堆新内容，而是控制模块边界。

当前已经存在的合理拆分：

- `config/*.json`：数值和内容配置。
- `js/*_state.js`：偏规则、状态、结算。
- `js/*.js`：偏 UI 渲染和交互。
- `js/game_systems.js`：统一系统调度。
- `js/main.js`：启动、读档、保存、离线结算、定时器。

当前仍然偏耦合的地方：

- `data.js` 职责太多，包含配置访问、查询、战斗属性、卡片效果、物品来源。
- `combat_state.js` 仍然混着旧手动战斗、无尽塔、区域战斗、关键威胁。
- `settleOffline()` 还在 `main.js`，并且在线收益与离线收益公式没有完全共用。
- 很多模块仍然直接读写全局 `state`，短期可接受，但要避免继续扩大影响范围。

## 二、后续新增玩法的标准结构

新增一个挂机玩法时，不要直接把逻辑写进 `main.js`。

推荐结构：

```text
config/xxx.json          玩法配置、数值、掉落、解锁条件
js/xxx_state.js          状态变更、结算、规则判断
js/xxx.js                页面渲染、按钮交互、UI 状态
js/game_systems.js       注册 tick / render / init
index.html               只增加必要 DOM 和脚本引用
```

新增玩法的基本步骤：

1. 在 `config-loader.js` 加载对应配置。
2. 在 `state_core.js` 增加默认存档结构和读档兼容。
3. 在 `xxx_state.js` 写规则，不写 DOM。
4. 在 `xxx.js` 写渲染和交互，不写复杂结算。
5. 在 `game_systems.js` 注册系统推进和定时渲染。
6. 在 `index.html` 增加页面结构和脚本。
7. 跑语法检查，并用浏览器验证页面启动无错误。

## 三、主循环规则

`main.js` 不应该再继续膨胀。

当前约定：

- `main.js` 负责启动、加载配置、读档、离线结算、保存、定时器。
- `game_systems.js` 负责在线 tick、定时 UI 刷新、统一初始化。
- 单个玩法的完成结算留在自己的 state 文件里。

新增在线挂机循环时，应该注册到 `GAME_TICK_SYSTEMS`，并尽量复用：

```js
advanceProgressLoop(...)
```

不要在 `main.js` 里手写新的：

```js
state.xxx.progress += dt / duration;
while (state.xxx.progress >= 1) { ... }
```

## 四、状态层与 UI 层边界

`*_state.js` 应该做：

- 判断是否可执行。
- 消耗材料。
- 发放物品、金币、经验。
- 修改存档状态。
- 返回结果对象。

`*.js` UI 文件应该做：

- 读取状态并渲染页面。
- 绑定按钮事件。
- 调用 state 层函数。
- 根据结果显示 toast 或刷新页面。

UI 层不要直接散落复杂结算公式。状态层也不要直接拼大量 HTML。

## 五、配置优先原则

能放进配置的内容，不要写死在代码里。

适合配置化：

- 采集点、鱼种、矿物、木材。
- 怪物、区域、关键威胁。
- 掉落、权重、经验、金币。
- 装备、配方、解锁条件。
- 技能基础时间、成功率、成长参数。

适合代码实现：

- 通用公式。
- 状态迁移。
- 权重抽取。
- 战斗模拟。
- UI 事件。

## 六、后续优先重构顺序

### 1. 抽离收益结算服务

建议新增：

```text
js/reward_service.js
```

目标是统一处理：

- 金币倍率。
- 经验倍率。
- 图鉴加成。
- 药水加成。
- 卡片加成与次数消耗。
- 日志格式。

钓鱼、伐木、采矿、宝石、战斗都应该逐步接入这套结算，避免每个系统各算一遍。

### 2. 抽离离线收益

建议新增：

```text
js/offline_state.js
```

`settleOffline()` 不应该长期留在 `main.js`。离线收益也不应该重新写一套收益公式，而应该调用在线收益的同一套结算服务，只是循环次数不同。

### 3. 拆分 `data.js`

建议拆成：

```text
js/data_access.js       配置读取、按 id 查找
js/combat_stats.js      战斗属性计算
js/source_lookup.js     物品来源说明
```

### 4. 拆分战斗状态

建议拆成：

```text
js/combat_styles_state.js
js/combat_area_state.js
js/combat_battle_sim.js
js/combat_legacy_state.js
```

战斗是后续会继续扩展的核心系统，不能一直把所有战斗逻辑放在一个文件里。

### 5. 建立通用解锁条件

建议所有系统逐步使用统一条件结构：

```json
{
  "requires": [
    { "type": "skillLevel", "skill": "mining", "level": 30 },
    { "type": "areaThreatCleared", "areaId": "mine_entrance" },
    { "type": "itemOwned", "id": "steel_pickaxe", "count": 1 }
  ]
}
```

这样采集点、配方、区域、装备、任务都能共用一套判断。

## 七、编码与验证要求

每次改代码后至少做：

```text
node --check js/改动文件.js
```

如果改动多个 JS 文件，做全量语法检查。

如果改了页面脚本、主循环、状态结构、战斗、存档，必须浏览器刷新验证：

- 页面能启动。
- loading 消失。
- 当前版本脚本已加载。
- 控制台没有当前版本的新错误。
- 对应玩法能正常推进一次。

## 八、后续和 Codex 协作时的默认指令

以后涉及新增玩法、重构、修系统性问题，可以直接说：

```text
按《项目技术方案.md》来做。
```

默认含义：

- 不把新玩法直接塞进 `main.js`。
- 按 `config + state + UI + game_systems` 的结构实现。
- 优先复用现有工具函数。
- 避免新增全局杂乱函数。
- 修改后做语法检查和浏览器验证。

## 九、当前保留的技术债

这些不是必须马上修，但后续要逐步处理：

- 旧战斗逻辑仍有残留，无尽塔与手动战斗需要重新评估。
- 中文注释和部分历史内容在某些终端里显示乱码，后续可以统一整理编码。
- `data.js` 和 `combat_state.js` 文件仍然偏大。
- 在线收益和离线收益仍未完全共用。
- 缺少配置校验脚本，后续内容变多后容易出现错误 ID、缺材料、重复 ID。

## 十、核心原则

新功能可以先小，但结构不能乱。

一个玩法能不能长期扩展，主要看三件事：

1. 配置是否清晰。
2. 状态结算是否独立。
3. 主循环是否只负责调度。

只要守住这三点，后面继续做锻造、炼金、农业、任务、成就、精通、地图探索，都不会把项目重新拖回一团乱。
