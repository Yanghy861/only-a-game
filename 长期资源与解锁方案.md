# 长期资源与解锁方案

更新时间：2026-06-28

这份文档用于约束后续长期开发。目标不是一次性把内容做满，而是保证每次新增玩法、资源、区域、装备时，都能接回同一条长期成长链。

## 一、长期项目的核心判断

这个项目不应该继续横向堆系统，而应该先形成稳定的循环：

```text
生活采集 -> 制作装备/消耗品 -> 区域战斗 -> 解锁新区域/新资源 -> 回到生活采集
```

每个新系统至少要回答三个问题：

1. 它产出什么？
2. 它消耗什么？
3. 它推进哪个系统？

如果一个系统只产出金币，长期会变成无意义挂机；如果一个材料没有消耗口，长期会变成库存垃圾。

## 二、当前版本的资源定位

| 系统 | 主要产出 | 当前定位 | 后续方向 |
| --- | --- | --- | --- |
| 钓鱼 | 鱼、水域材料、少量特殊掉落 | 食物与饰品辅助材料 | 主要服务烹饪、战斗饰品、任务提交 |
| 伐木 | 木材 | 生活工具、部分武器/副手 | 继续作为工具、弓、法杖、护符基础材料 |
| 采矿 | 矿石 | 生活工具、金属装备基础 | 继续作为武器、防具、强化材料主线 |
| 宝石 | 原石、成品宝石 | 卡片与后续装备强化材料 | 后续接入戒指、项链、镶嵌、炼金 |
| 战斗 | 怪物材料、区域推进 | 装备制作与世界解锁 | 作为长期内容门槛和材料来源 |
| 卡片 | 次数型加成 | 短期消耗品 | 后续作为跨系统“爆发按钮”，不做永久堆数值 |
| 烹饪 | 食物 buff | 目前仍偏手动 | 后续改成持续加工/自动消耗补给 |

## 三、解锁链路原则

采集等级只代表熟练度，不应该独自决定所有内容是否可见。长期项目需要“等级 + 世界推进”共同决定。

推荐规则：

```json
{
  "requiredLevel": 30,
  "unlock": { "type": "threatCleared", "areaId": "mine_entrance" }
}
```

当前已采用的最小规则：

| 内容 | 解锁逻辑 |
| --- | --- |
| 基础钓鱼、基础伐木、露天矿坑、河床宝石 | 开局可用 |
| 浅层矿洞 | 清理林地边缘关键威胁后开放 |
| 深层矿洞、银矿脉、宝石矿脉 | 清理矿洞入口关键威胁后开放 |
| 远洋及更深钓点、黄金矿脉、深渊宝石洞 | 清理潮湿洞穴关键威胁后开放 |

以后新增区域时，不要直接把全部后续区域显示给玩家。应该只显示：

- 已发现区域
- 当前威胁
- 一条传闻
- 未知占位

## 四、资源消耗原则

每类资源都应该有至少一个稳定消耗口。

| 资源类型 | 推荐消耗口 |
| --- | --- |
| 普通鱼 | 烹饪、卡片、任务提交 |
| 高级鱼 | 高级烹饪、Boss 前补给、稀有任务 |
| 水域材料 | 戒指、项链、护符、炼金 |
| 木材 | 工具、弓、法杖、副手、生活设施 |
| 矿石 | 武器、防具、工具、强化 |
| 宝石 | 饰品、卡片、装备镶嵌 |
| 怪物材料 | 战斗装备、区域钥匙、特殊卡片 |

当前已做的最小调整：

- 部分戒指/项链开始消耗 `shell`、`pearl`、`fish_bone`、`sea_salt`、`coral_shard`、`sealed_bottle`。
- 采集点支持统一 `unlock` 条件，后续可以继续扩展到配方、任务、商店、Boss。

## 五、后续优先级

### P0：长期骨架

1. 存档安全。
2. 资源流转与解锁链路。
3. 配置校验脚本，检查缺失材料、无用途材料、重复 ID。
4. 离线收益与在线收益共用结算。

### P1：核心挂机体验

1. 烹饪改成持续加工，不再是点一下生成再点一下吃。
2. 战斗补给规则确定：每场满状态，食物/卡片作为额外策略资源。
3. 区域战斗掉落和装备制作进一步平衡。

### P2：内容扩展

1. 扩展到 10-15 个关键威胁。
2. 新增更多区域，但按传闻逐步揭示。
3. 加任务/成就/长期收藏目标。

### P3：深度系统

1. 装备强化、镶嵌、套装。
2. 炼金、农业、建筑等长期材料消耗系统。
3. 赛季或挑战模式。

## 六、以后新增内容的检查清单

新增任何材料前先检查：

- 是否有来源？
- 是否有消耗？
- 是否属于某个阶段？
- 是否会挤占旧材料价值？

新增任何区域前先检查：

- 它由哪个威胁或条件解锁？
- 它产出的材料用于哪一批装备/食物/卡片？
- 它是否只提前暴露一条传闻？

新增任何装备前先检查：

- 是否消耗至少两类系统资源？
- 是否提供一种构筑倾向，而不是纯数值碾压？
- 是否会让上一阶段装备完全失去意义？

长期目标：每次更新都让“旧系统多一个理由继续存在”，而不是只给玩家多一个按钮。
